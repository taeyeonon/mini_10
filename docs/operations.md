# 운영 및 테스트 가이드

## 환경변수

| 이름 | 필수 범위 | 설명 |
|---|---|---|
| `TOSS_CLIENT_KEY` | 선택 | 브라우저 결제창용 기본 테스트 키 덮어쓰기 |
| `TOSS_SECRET_KEY` | 결제 테스트 | 서버 승인용 시크릿 키 |

공유 테스트 클라이언트 키는 기본 설정되어 있습니다. 시크릿 키는 로그, 화면, README, Git 기록에 남기지 않으며 테스트 키와 운영 키를 분리합니다.

## 기본 시연 시나리오

1. 관리자 로그인 → 회원 선택 → 수강권 수동 발급
2. 트레이너 로그인 → 반복 일정 등록 → 실제 수업 생성 확인
3. 회원 로그인 → 수강권 또는 토스 테스트 결제 → 수업 예약
4. 회원 예약 취소 → 수강권 복구 확인
5. 트레이너 일정 취소·복원 및 새로고침 정리 확인

## 결제 테스트

- 현재 가격은 시연용 1회당 100원입니다.
- 브라우저 금액이 아니라 서버가 `ticketCount × 100`으로 계산합니다.
- 성공 URL의 금액, DB 주문 금액, 토스 승인 결과 금액이 모두 일치해야 지급합니다.
- 이미 `PAID`인 주문은 다시 승인하거나 수강권을 중복 지급하지 않습니다.
- 실제 토스 키는 `application.properties`에 직접 작성하지 않습니다.

## 데이터 초기화

- `ADMIN`, `TRAINER`, `CUSTOMER` 역할은 없을 때만 생성합니다.
- 샘플 사용자는 이메일 존재 여부를 확인한 후 생성합니다.
- 기존 계정의 비밀번호나 역할을 자동으로 덮어쓰지 않습니다.
- 결제·예약·일정 데이터는 자동으로 삭제하거나 재생성하지 않습니다.

## Git 협업 흐름

1. `develop` 최신 변경을 pull/fetch 합니다.
2. 개인 기능 브랜치에서 구현하고 테스트합니다.
3. 1차 MVP는 `develop`에 병합해 통합 테스트합니다.
4. 릴리스 가능한 상태에서 `main`으로 병합합니다.
